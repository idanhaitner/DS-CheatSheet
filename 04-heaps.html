4.1 Binary Heap (Priority Queue)

About the heap

A complete binary tree in an array. Max-heap: parent ≥ children. Root = max. Height \(\Theta(\log n)\).

Index math (0-based) 
parent(i) = (i - 1) / 2
left(i)   = 2*i + 1
right(i)  = 2*i + 2
OperationTime
findMaxO(1)
insert / extractMaxO(log n)
increaseKey / maxHeapifyO(log n)
buildHeapO(n)
buildHeap is O(n): most nodes sit near leaves: not \(O(n\log n)\). Exam: no O(log n) search for an arbitrary key (that's O(n)).

4.2 Huffman Coding

Optimal prefix-free code for data compression: frequent characters get shorter codes. Build a binary tree bottom-up with a min-heap.

  1. Make a leaf for each character with its frequency; put all in a min-heap.
  2. Repeatedly extractMin the two smallest, create a parent with their summed frequency, insert it back.
  3. Repeat until one tree remains. Left edge = 0, right edge = 1; a character's code = path from root.
PhaseTimeWhy
Build tree (n distinct symbols)O(n log n)n−1 merges; each merge = 2 extractMin + 1 insert on a min-heap of size O(n)
Build code tableO(n)One DFS/BFS from root to label each leaf with its 0/1 path
Encode message (length N)O(N)Look up each character's code in O(1) (hash table / array)
Decode messageO(N)Walk the tree one bit at a time; each source symbol read once
SpaceO(n)Tree has 2n−1 nodes; code table has n entries

Expected code length: \(\sum_c freq(c)\cdot depth(c)\): weighted path length; Huffman minimizes this among all prefix codes. An optimal Huffman tree is a full binary tree; the two least-frequent symbols are siblings at the deepest level.

The input \(C\) is a list of characters and their frequencies.

pseudocode 
Huffman(C):
    n = |C|
    Q = priority queue of C
    for i = 1 to n - 1:
        allocate a new node z
        z.left = ExtractMin(Q)
        z.right = ExtractMin(Q)
        z.freq = z.left.freq + z.right.freq
        Insert(Q, z)
    return ExtractMin(Q)

4.3 Lempel-Ziv Compression

Unlike Huffman (which needs frequencies in advance), Lempel-Ziv is a dictionary-based method that learns repeated substrings on the fly, replacing repeats with short references. Behind gzip, ZIP, PNG.

  • Maintain a growing dictionary of substrings seen so far.
  • Find the longest prefix already in the dictionary; output its index, then add (prefix + next char) as a new entry.
  • Later repetitions become a single compact index instead of raw characters.

Example: encode A B A B A B A

StepOutput tokenNew dictionary entry
read A (new)(0, A)1 = "A"
read B (new)(0, B)2 = "B"
read AB (A known)(1, B)3 = "AB"
read ABA (AB known)(3, A)4 = "ABA"

Time complexity

PhaseTimeWhy
Encode (naive dictionary scan)O(N²)≤ N steps; each step may scan all O(N) dictionary entries and compare prefixes
Encode (with trie / efficient match)O(N)One left-to-right pass; trie lookup per step is O(match length), totals O(N)
DecodeO(N)One pass; each token outputs ≥ 1 new character and adds one dictionary entry
SpaceO(N)Dictionary grows to O(N) phrases in the worst case
Huffman vs Lempel-Ziv:
  • Huffman: statistical, per-symbol, needs frequencies, optimal prefix code for a known distribution. Build: O(n log n).
  • Lempel-Ziv: dictionary, exploits repeated substrings, adaptive/online (one pass), decoder rebuilds the same dictionary. Encode: O(N²) naive / O(N) with trie; decode: O(N).
Both are lossless. Real tools (DEFLATE in gzip/PNG) combine LZ + Huffman.
