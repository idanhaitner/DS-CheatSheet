8.3 Exam Strategy & Common Pitfalls

How the exam asks questions (from past papers)

  • Warm-up asymptotics / Master Theorem; a large always / sometimes / never block; then one or two design-a-structure questions (often with a cheaper partial-credit variant). Default analysis is worst-case; use expected / amortized only when stated. Cite lecture theorems by name.
  • "Best time to implement operation X on structure Y": know each structure's operation costs cold. E.g. skip-list Delete (with a pointer) is \(\Theta(1)\) expected because \(E[\text{node height}]=1\); Search without a pointer is \(\Theta(\log n)\) expected / \(O(n)\) worst.
  • Sorting best/worst fill-ins: given constraints (e.g. each element within \(n^{0.4}\) of its index), work out Insertion/Merge/Quick/Counting behavior. Insertion is O(n·shift-distance); deterministic Quick's worst is on sorted input. For linear sorts, check whether \(k=O(n)\) or \(d=o(\log n)\): see §2.3 decision guide. Mixed constraints (most keys in a poly(\(n\)) range, few outliers): split → linear-ish sort + comparison sort → merge.
  • "Best algorithm to sort \(n\) integers with \(L\) binary digits": compare Radix \(\Theta(L\cdot n)\) vs Merge \(\Theta(n\log n)\). Radix when \(L=o(\log n)\); Merge when \(L=\Omega(\log n)\).
  • Design a structure meeting target complexities: first write the fields (what is stored where), then each op in 3–5 lines, then the bound. Combine primitives: hash (O(1) membership) + AVL / skip list / heap (order, Select, DeleteMax); OST + size/sum for Rank / Select / prefix sums; two heaps or dual sorted halves when Median / SelectN must be \(O(1)\) under a size bound. Hard ops (SumValues, fast Select for small \(i\)) are where full points live.
  • Amortized design: growing binary counter (\(f(k)=2k\) vs \(k+1\)); token/accounting proofs; Insert amortized \(O(1)\) via periodic mass rebuilds / deletes. See §7 amortized.
  • Compression: LZ78 (shortest / longest string with a fixed number of blocks); Huffman vs optimal prefix codes (two rarest symbols need not be siblings in every optimal tree). See Huffman · LZ.
  • Graph counting / proof: e.g. “number of distinct BFS trees”, "is \(e_{\min}\) in every MST?", "is edge \(e\) in some MST?": see §6.9 graph exam tips. DFS back edges & discovery times; connected undirected DFS forest has one tree; BFS tree depends on neighbor order.
  • Graph reductions: super-source/sink, vertex splitting, reverse graph, reduce to MST. Always define \(G'\) and analyze build + algorithm time.
  • Union-Find shape questions: union by rank + path compression; when ranks are equal, know the lecture tie-break (which root becomes parent).

Top pitfalls to avoid

  • buildHeap is O(n), not O(n log n). Heapsort overall is O(n log n).
  • Heap order ≠ BST order: “every left-subtree key < every right-subtree key” is never required. Small values are often leaves, not always for every \(n\).
  • Quicksort worst = O(n²) (deterministic pivot on sorted input); average = O(n log n). Sorted → always \(\Theta(n^2)\); “not sorted ascending/descending” → \(\Theta(n^2)\) only sometimes.
  • Counting sort is O(n+k): only linear when \(k = O(n)\); it is stable, radix depends on that. Without a bound on max key \(K\), you cannot sort in \(\Theta(N)\) with counting/radix.
  • Radix sort is \(\Theta(d(n+b))\): needs stable digit sort. With \(L\) binary digits and 1-bit passes: \(\Theta(L\cdot n)\). Compare to merge before answering "best sort".
  • Bucket sort average = O(n) but worst = O(n²) when all elements fall in one bucket (true even without a uniform-[0,1] promise); \(\Theta(n)\) worst-case does not hold.
  • Dijkstra breaks on negative edges: use Bellman-Ford.
  • BFS ≠ shortest path on weighted graphs: only unweighted hop count. With all weights \(=1\), BFS distances match shortest paths, but the tree/path need not equal Prim’s MST path.
  • Spanning tree ≠ MST: MST additionally minimizes total weight; cycle/cut properties for proofs.
  • Plain BST is O(n) worst: only AVL/balanced guarantees O(log n). You can still build a BST from a sorted array in \(O(n)\) by picking medians (not by left-to-right inserts).
  • Hash chaining: insert always O(1); search Θ(1+α) expected, O(n) worst. DLL vs AVL in each cell changes per-bucket search, not global Successor. Probing: expected O(1) probes, worst O(n); needs α<1. Tombstones (delete method 1) inflate \(\alpha'\) vs \(\alpha\); method 2 reinserts the probe block.
  • AVL-per-bucket hash improves worst-case search in a cell to \(O(\log n)\), but does not give global Successor in \(O(\log n)\) by itself (must know which bucket / scan buckets).
  • Huffman’s algorithm always yields an optimal prefix code for that algorithm; optimal prefix trees in general need not make the two rarest symbols siblings.
  • Master Theorem gap: doesn't apply when f differs by only a log factor.
  • Θ vs O: give Θ for a tight bound; O suffices for an upper bound unless stated.
  • State which theorem you invoke (lecture results allowed, homework results not).
