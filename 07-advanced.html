7.1 Amortized Analysis

Average cost per operation over a worst-case sequence: not a probabilistic average. Guarantees any sequence of \(m\) operations costs at most \(m\times(\text{amortized cost})\).

MethodIdeaExample
AggregateTotal cost of m ops is \(T(m)\); amortized = \(T(m)/m\).Dynamic array (doubling): n insertions copy at most n + n/2 + n/4 + … < 2n elements total → \(T(n)=O(n)\) ⟹ O(1) amortized per insert.
AccountingAssign each op a charge \(\hat c_i\); cheap ops overpay to bank credit. Require bank ≥ 0: \(\sum\hat c_i\ge\sum c_i\).Same dynamic array: charge 3 coins per insert: 1 pays the insert, 2 go to the bank. When the array doubles (cost n), pay from the bank (each of the n/2 elements since last resize saved 2 coins). Bank never goes negative ⟹ O(1) amortized.

Classic results

  • Dynamic array (doubling): n insertions cost O(n) total ⟹ O(1) amortized per insert.
  • Stack with Multipop: any sequence of push/pop/multipop on n elements is O(n) total ⟹ O(1) amortized (each element pushed once, popped at most once).
  • Binary counter increment: n increments flip O(n) bits total ⟹ O(1) amortized per increment.

7.2 Union-Find (Disjoint Sets)

Maintains a partition into disjoint sets, each with a representative. Backbone of Kruskal's MST and connectivity queries.

OperationMeaning
MakeSet(x)new singleton set {x}
Find(x)return representative of x's set
Union(x,y)merge the two sets

Forest implementation + two optimizations

  • Union by rank/size: attach the smaller tree under the larger root → height O(log n).
  • Path compression: during Find, point every node on the path directly to the root.
ConfigurationCost of m operations
Linked list + weighted unionO(m + n log n)
Forest, no optimizationO(m·n) worst
Both optimizationsO(m·α(n)) ≈ O(m): α = inverse Ackermann, ≤ 4 in practice
Application: connected components: MakeSet each vertex, then for each edge (u,v) Union their sets. Two vertices are connected ⟺ same Find.

7.3 Deamortization of rehashing

Problem: when a hash table grows, a normal rehash copies all \(n\) keys in one Insert → \(\Theta(n)\) spike. Deamortization spreads that copy over many Inserts so every Insert stays \(\Theta(1)\) worst-case.

Setup (chaining)

Table \(T\) has \(m\) cells \(T[0],\ldots,T[m-1]\). Each cell is the head of a linked list of keys that hashed there. There are \(n\) keys total; we keep \(n/m\in\Theta(1)\).

  • When \(n=m\): grow to size \(2m\) (copy every key into a new table).
  • When \(n=m/4\): shrink to \(m/2\) (slides focus on growing).

What a full rehash does

Copy from \(T\) (size \(m\)) into a new table \(T'\) (size \(2m\)) by walking every cell and every key:

Rehash — full copy 
(1) create T′ of size 2m
(2) for i = 0 to m−1
(3)     p ← T[i]              // open cell i
(4)     while p ≠ NULL
(5)         Insert(T′, p)     // copy this key
(6)         p ← p.next
Count the tiny steps in that loop. The lecture treats the copy as \(2m\) atomic actions (call them “copy steps”):
  • \(m\) cell opens: once per index, \(p\leftarrow T[i]\) (even if the list is empty).
  • \(m\) key copies: once per key, Insert(T′, key) then \(p\leftarrow p.next\).
Doing all \(2m\) of them inside one Insert costs \(\Theta(n)\). Deamortization does only a constant number of them per Insert.

Deamortized Insert

Keep two tables: live \(T\) (size \(m\)) and building \(T'\) (size \(2m\)). Remember where you paused in the copy: cell index \(i\) and list pointer \(p\).

Define one helper that does exactly one tiny action from the rehash loop:

CopyOneStep — open a cell OR copy one key 
CopyOneStep():
    if p = NULL:                 // finished the current list
        p ← T[i]                  // open the next cell
        i ← i + 1
    else:
        Insert(T′, p)             // copy this one key into T′
        p ← p.next

Then Insert only calls that helper a constant number of times:

Insert(x) 
Insert(x):
    insert x into T

    if h(x) ≥ i:                 // scan has not reached x’s cell yet
        call CopyOneStep() 4 times
    else:                       // scan already passed x’s cell
        insert x into T′ as well  // otherwise T′ would miss x
        call CopyOneStep() 3 times

    if |T| = m:                  // copy finished; switch tables
        T ← T′
        T′ ← new empty table of size 4m   (smart array → Θ(1))
Example (\(m=4\), keys \(1,8\) in \(T\), Insert(4)). After putting 4 into \(T\), the four CopyOneStep calls might be:
  1. \(p\leftarrow T[0]\) (open cell 0 → list \(1\to 8\))
  2. Insert(T′, 1); \(p\leftarrow\) next
  3. Insert(T′, 8); \(p\leftarrow\) NULL
  4. \(p\leftarrow T[1]\) (open cell 1, empty)
Stop. Next Insert continues from cell 2.
Why every Insert is \(\Theta(1)\)
Put \(x\) in \(T\) (and maybe \(T'\))\(\Theta(1)\)
At most 4 calls to CopyOneStep\(\Theta(1)\)
When \(n=m\): swap + smart-array allocate\(\Theta(1)\)

Why 4? Total copy work \(2m\) steps / \(m/2\) Inserts = 4 steps per Insert.

Smart array: allocating size \(4m\) without zeroing every cell costs \(\Theta(1)\) in the course model.

7.4 B-Tree Amortized Analysis

For a B-tree with 2-pass insertion/deletion, split/merge/shift operations can be charged to individual inserts/deletes.

Theorem

Starting from an empty tree, in any sequence of \(m\) insertions and deletions, the number of merge, split, and shift operations is at most \(\frac{3}{2}m\).

Accounting method

  • Cost of each merge / split / shift = 1.
  • Charge per insert or delete = \(3/2\).

Invariant (dollar storage on nodes):

  • Non-root node with \(t-1\) keys stores \(1/2\) dollar.
  • Node with \(2t-1\) keys stores \(3/2\) dollars.

Insertion: when node \(v_i\) splits (had \(2t-1\) keys), use its \(3/2\) dollars to pay for the split and put \(1/2\) on each child. If parent \(w\) reaches \(2t-1\) keys, take \(3/2\) from the operation's charge.

Deletion: when two siblings merge, use \(1/2\) dollar from each to pay the merge. Shifts funded from the \(3/2\) charge.

Amortized cost per insert/delete = O(1) for structural ops (tree height still \(O(\log n)\) per search). See B-trees for the structure itself.
