Insertion Sort O(n²)

Idea. Grow a sorted prefix on the left. Take the next "key" and shift larger sorted elements right until the key lands in place: like sorting cards in your hand.

pseudocode 
for j = 1 to n-1:
    key = A[j]
    i   = j - 1
    while i >= 0 and A[i] > key:
        A[i+1] = A[i]        # shift right
        i = i - 1
    A[i+1] = key             # drop key into the gap
Best (already sorted)Θ(n)
Average / Worst (reverse)Θ(n²)
SpaceO(1), in-place, stable
Exam gold: runs in O(n·d) where d = number of inversions, or O(nk) when every element is ≤ k positions from its final spot. Best for nearly-sorted or small arrays.

Selection Sort O(n²)

Idea. Repeatedly find the minimum of the unsorted suffix and swap it to the front. Always \(\Theta(n^2)\) even if already sorted, but does only n−1 swaps.

pseudocode 
for i = 0 to n-2:
    min = i
    for j = i+1 to n-1:
        if A[j] < A[min]:
            min = j
    swap(A[i], A[min])
Useful when writes are expensive (only n−1 swaps total). Not stable.

Bubble Sort O(n²)

Idea. Repeatedly walk the array swapping adjacent out-of-order pairs; the largest element "bubbles" to the end each pass. With an early-exit flag it is \(\Theta(n)\) on sorted input.

pseudocode 
for i = 0 to n-2:
    swapped = false
    for j = 0 to n-2-i:
        if A[j] > A[j+1]:
            swap(A[j], A[j+1])
            swapped = true
    if not swapped:
        break                # already sorted, stop early

Best Θ(n) · Average/Worst Θ(n²) · O(1) space, stable.

Merge Sort O(n log n)

Idea. Divide the array in half, recursively sort each half, then merge the two sorted halves in linear time. \(T(n)=2T(n/2)+\Theta(n)\Rightarrow \Theta(n\log n)\) in all cases.

pseudocode 
MergeSort(A, lo, hi):
    if lo < hi:
        mid = (lo + hi) / 2
        MergeSort(A, lo, mid)
        MergeSort(A, mid+1, hi)
        Merge(A, lo, mid, hi)     # O(n): two pointers, copy smaller

Quick Sort avg n log n worst n²

Idea. Pick a pivot, partition so smaller elements go left and larger go right (pivot lands in its final spot), then recurse on each side. In-place, small constants.

partition (Lomuto, pivot = last) 
Partition(A, lo, hi):
    pivot = A[hi]
    i = lo - 1
    for j = lo to hi-1:
        if A[j] <= pivot:
            i = i + 1
            swap(A[i], A[j])
    swap(A[i+1], A[hi])          # put pivot in place
    return i+1
Best / AverageΘ(n log n)
Worst (sorted + last-pivot)Θ(n²)
Worst case occurs when the pivot is always min/max → splits of size (n−1, 0): \(T(n)=T(n-1)+\Theta(n)=\Theta(n^2)\). A deterministic last-element pivot triggers this on already-sorted input. Fix with RandomizedPartition (or median-of-three).

Heap Sort O(n log n)

Idea. Build a max-heap in O(n). Then repeatedly swap the root (max) with the last element, shrink the heap, and sift the new root down. In-place, guaranteed \(\Theta(n\log n)\).

pseudocode 
HeapSort(A):
    BuildMaxHeap(A)              # O(n)
    for i = n-1 downto 1:
        swap(A[0], A[i])         # move current max to the end
        heapSize = heapSize - 1
        MaxHeapify(A, 0)         # O(log n)
Advantage over merge sort: O(1) extra memory. Advantage over quicksort: guaranteed O(n log n). See Heaps for the heapify diagram.

2.3 Linear Sorting Algorithms

Linear sorting = non-comparison sorts that can beat the \(\Omega(n\log n)\) lower bound by exploiting structure on the keys (bounded integer range, fixed digits, uniform distribution). They are not comparison-based.

Quick comparison

AlgorithmTimeWhen it is truly \(\Theta(n)\)Key requirementStable?Extra space
Counting Θ(n + k) \(k = O(n)\) Integers in \([0, k]\) Yes O(n + k)
Radix Θ(d(n + b)) \(d = O(1)\) and \(b = O(n)\) (so \(d(n+b)=O(n)\)) \(d\) digits in base \(b\); digit sort must be stable Yes O(n + b)
Bucket Θ(n) avg Uniform on \([0,1)\), ~1 element per bucket Inputs uniformly distributed in \([0,1)\) Yes O(n)

\(k\) = key range upper bound · \(d\) = number of digits · \(b\) = radix base (often 10 or 256; for single-bit passes \(b=2\)).

Which linear sort to choose?

ConditionPickWhy
Integers in \([0, 2N]\) or \([0, cN]\) for constant \(c\) Counting sort \(k = O(N)\) ⇒ \(\Theta(N + k) = \Theta(N)\). Direct-address table of size \(2N+1\) is enough.
Integers in \([0, N^2]\), need time \(\Theta(N)\) Radix sort (base \(N\)) In base \(N\), each key has 2 digits (each in \(\{0,\ldots,N-1\}\)). Two counting passes: \(\Theta(2(N+N))=\Theta(N)\).
Each \(A[i]\) is within \(n^{0.4}\) of its sorted index (\(i \le A[i] \le i + n^{0.4}\)) Counting sort \(\Theta(n)\) Max key \(\le n + n^{0.4} = O(n)\), so \(k = O(n)\). Insertion is only \(\Theta(n^{1.4})\) here.
Each number has exactly \(L\) binary digits; \(L = n\) Merge sort \(\Theta(n\log n)\) Radix with 1-bit passes costs \(\Theta(L\cdot n) = \Theta(n^2)\): worse than merge.
Each number has exactly \(L\) binary digits; \(L = (\log n)^{1/2}\) Radix sort \(\Theta(n(\log n)^{1/2})\) \(L\cdot n = n(\log n)^{1/2} = o(n\log n)\). Radix beats merge.
Each number has exactly \(L\) binary digits; \(L = (\log n)^{3/2}\) Merge sort \(\Theta(n\log n)\) \(L = (\log n)^{3/2} = \omega(\log n)\), so 1-bit radix costs \(\Theta(n(\log n)^{3/2})\): worse than merge.
Each number has exactly \(L\) binary digits; \(L = (\log n)^3\) Merge sort \(\Theta(n\log n)\) \(L\cdot n = n(\log n)^3 \gg n\log n\). Comparison sort wins.
Salaries / weights with fixed digit representation (e.g. 10 digits, base 10) Radix sort \(d\) and \(b\) are constants ⇒ \(\Theta(d(n+b)) = \Theta(n)\). Use counting sort per digit.
Keys uniformly distributed in \([0, 1)\) Bucket sort \(n\) buckets, insertion sort each ⇒ \(\Theta(n)\) expected. Worst \(\Theta(n^2)\) if all land in one bucket.
Need to sort list \(L\) in \(\Theta(N)\) but no bound on \(K\) (max key) Impossible in \(\Theta(N)\) Comparison lower bound \(\Omega(N\log N)\). Do not use counting/radix; use merge/heap/AVL insert-all (\(\Theta(N\log N)\)).

Radix vs Merge: the one comparison to memorize

Radix (with counting per digit, base \(b\)): \(\Theta(d\cdot(n+b))\). Merge: \(\Theta(n\log n)\).

  • Radix wins when \(d\cdot n = o(n\log n)\), i.e. \(d = o(\log n)\).
  • Merge wins when \(d = \omega(\log n)\) (e.g. \(d = (\log n)^{3/2}\), \(d = n\), or \(d = (\log n)^3\)). If \(d = \Theta(\log n)\), both are \(\Theta(n\log n)\).
  • Counting wins over radix when there is a single digit (one pass) and \(k = O(n)\).
Top traps for linear sorts:
  • Counting is \(\Theta(n+k)\). If \(k = n^2\), both time and space are \(\Theta(n^2)\) — not linear. For range \([0,n^2]\) with \(\Theta(n)\) time, use radix (base \(n\), \(d=2\)).
  • Radix requires a stable digit sort (counting sort works). Without stability, radix breaks.
  • Bucket sort worst case is \(\Theta(n^2)\): never claim worst-case \(\Theta(n)\) without the uniform-distribution assumption.
  • Don't confuse linear probing (hashing) with linear sorting (counting/radix/bucket).
Design pattern: bounded range \([0,cN]\) ⇒ counting sort + direct-address array; fixed digits ⇒ radix; uniform floats ⇒ bucket; unbounded keys ⇒ comparison sort or AVL, never counting/radix alone.

2.3.1 Counting Sort O(n + k)

Idea (non-comparison). For integer keys in \([0,k]\): count occurrences, take prefix sums to know final positions, then place elements right to left (preserves stability).

pseudocode 
CountingSort(A, k):
    C = new int[k+1] filled with 0
    for a in A:          C[a] = C[a] + 1        # counts
    for i = 1 to k:      C[i] = C[i] + C[i-1]   # prefix sums
    for j = n-1 downto 0:                       # right to left = stable
        C[A[j]] = C[A[j]] - 1
        B[C[A[j]]] = A[j]

2.3.3 Bucket Sort avg O(n)

Idea. For inputs uniform over \([0,1)\): split into n equal buckets, drop each element into bucket \(\lfloor n\cdot x\rfloor\), sort each bucket (insertion sort), then concatenate.

pseudocode 
BucketSort(A):
    create n empty lists B[0..n-1]
    for x in A:
        append x to B[floor(n * x)]
    for i = 0 to n-1:
        insertionSort(B[i])
    return concat(B[0..n-1])

Average Θ(n) (uniform input) · Worst Θ(n²) (all in one bucket).

The Comparison-Sort Lower Bound

Any comparison-only sort is a decision tree: internal nodes are comparisons, leaves are permutations. There are \(n!\) orderings, so \(\ge n!\) leaves. A binary tree with \(L\) leaves has height \(\ge \log_2 L\), so worst-case comparisons \(\ge \log_2(n!) = \Theta(n\log n)\).

Conclusion: Merge/Heap sort are asymptotically optimal comparison sorts. Counting/Radix/Bucket beat this only by not being comparison-based.

2.4 The Selection Problem

Input: array \(A\) of \(n\) distinct numbers and an integer \(i\).
Goal: return the \(i\)-th smallest element of \(A\) (the \(i\)-th order statistic).

  • \(i = 1\) → minimum (easy linear scan).
  • \(i = n\) → maximum.
  • \(i = \lceil n/2\rceil\) → median.

Sorting then indexing costs \(O(n\log n)\). Selection can be done in linear time: expected \(\Theta(n)\) with randomization, or worst-case \(\Theta(n)\) with median-of-medians.

2.4.1 Partition

Same primitive as in Quicksort. Pivot starts at \(A[r]\). After Partition, the pivot sits at index \(p\) and:

  • every element in \(A[l..p-1]\) is \(\le\) pivot,
  • every element in \(A[p+1..r]\) is \(>\) pivot.

During the scan there are four zones:

ZoneMeaning
\(A[r]\)pivot (moved into place at the end)
\(A[j..r-1]\)not yet examined
\(A[i+1..j-1]\)elements \(>\) pivot
\(A[l..i]\)elements \(\le\) pivot
Partition (Lomuto) 
Partition(A, l, r):
    i = l - 1
    for j = l to r - 1:
        if A[j] <= A[r]:
            i = i + 1
            swap(A[i], A[j])
    swap(A[i + 1], A[r])
    return i + 1                 # final index of the pivot

Time \(\Theta(n)\) for a subarray of length \(n = r-l+1\). After Partition, the pivot’s rank in the subarray is \(k = p - l + 1\) (number of elements in \(A[l..r]\) that are \(\le\) pivot).

2.4.2 Randomized Partition

A fixed pivot (e.g. last element) can always be the min/max → terrible splits. RandomizedPartition picks a uniform random index in \([l,r]\), swaps it to the end, then runs ordinary Partition.

pseudocode 
RandomizedPartition(A, l, r):
    i = Random(l, r)             # uniform in [l..r]
    swap(A[i], A[r])
    return Partition(A, l, r)

Used by both Randomized-Quicksort and Randomized-Select. For any fixed distinct input, expected Quicksort time becomes \(\Theta(n\log n)\) (randomness is in the algorithm, not the input).

Randomized-Quicksort: same as Quicksort, but line (2) calls RandomizedPartition instead of Partition.

2.4.3 Randomized-Select

Like Quicksort’s divide step, but recurse into only one side: the side that still contains the \(i\)-th element.

pseudocode (lecture) 
RandomizedSelect(A, l, r, i):   # i-th smallest in A[l..r], 1-based
    if l == r:
        return A[l]
    p = RandomizedPartition(A, l, r)
    k = p - l + 1                # rank of pivot in this subarray
    if i == k:
        return A[p]
    elif i < k:
        return RandomizedSelect(A, l, p - 1, i)
    else:
        return RandomizedSelect(A, p + 1, r, i - k)

Call: RandomizedSelect(A, 0, n-1, i) for the \(i\)-th smallest in the whole array.

When going right, subtract \(k\): the left side plus the pivot account for the \(k\) smallest elements in the current range, so the target becomes the \((i-k)\)-th in the right side.

2.4.4 Expected-time analysis

Same green/red coloring idea as in the randomized Quicksort analysis:

  • A node \(v\) is green if it is the root, or \(\mathrm{size}(v) \le \tfrac34 \cdot \mathrm{size}(\mathrm{parent})\). Otherwise red.
  • A green node has at most one red child; \(\Pr[\text{red child}] \le 1/2\).
  • For green \(v\): \(E[\mathrm{cost}(v)] \le 2\cdot\mathrm{size}(v)\).

Along the unique path of green nodes \(u_0,u_1,\ldots\) in a Select recursion tree:

\[ \mathrm{size}(u_i) \le \Bigl(\tfrac34\Bigr)^i n \qquad\Rightarrow\qquad E\Bigl[\sum_i \mathrm{cost}(u_i)\Bigr] \le 2\sum_i\Bigl(\tfrac34\Bigr)^i n = O(n). \]

So expected time of Randomized-Select is \(\Theta(n)\). Worst case is still \(O(n^2)\) if every pivot is extreme (low probability).

2.4.5 Deterministic Select (median of medians)

Same skeleton as Randomized-Select, but the pivot is chosen by ChoosePivot so every split is “good enough” in the worst case.

Select 
Select(A, l, r, i):
    if l == r: return A[l]
    j = ChoosePivot(A, l, r)
    swap(A[j], A[r])
    p = Partition(A, l, r)
    k = p - l + 1
    if i == k:     return A[p]
    elif i < k:  return Select(A, l, p - 1, i)
    else:          return Select(A, p + 1, r, i - k)

ChoosePivot (groups of 5)

  1. If \(n = r-l+1 \le 10\), return \(r\) (small case).
  2. Split \(A[l..r]\) into \(\lceil n/5\rceil\) groups of 5 (last group may be smaller).
  3. Sort each group; copy the group medians into array \(B\).
  4. \(x = \mathrm{Select}(B,\, \lceil |B|/2\rceil)\) — the median of medians.
  5. Return the index of \(x\) in \(A\).

Why the pivot is good

At least roughly half of the group-medians are \(\ge x\), and each such group contributes at least 3 elements \(\ge x\). Same for elements \(\le x\). Lecture bound:

\[ \#\{ \text{elements } > x \} \;\ge\; \tfrac{3}{10}n - 6 \qquad\text{(and likewise for }{<}\,x\text{)}. \]

So the recursive Select call works on at most \(\tfrac{7}{10}n + 6\) elements.

Recurrence

\[ T(n) = T\bigl(\lceil n/5\rceil\bigr) + T\bigl(\tfrac{7}{10}n + 6\bigr) + \Theta(n). \]

Simplified: \(T(n) = T(n/5) + T(7n/10) + n\). By induction \(T(n) \le cn\) for large enough \(c\) (since \(\tfrac15+\tfrac{7}{10}=\tfrac{9}{10}<1\)).

Worst-case \(\Theta(n)\).

Why groups of 5? Odd size → clean median. Fractions must sum to less than 1. Groups of 3 give \(T(n/3)+T(2n/3)+O(n)=O(n\log n)\): not linear.
