6.0 The Big Picture — Which Graph Algorithm?

Before memorizing pseudocode, ask: what is the graph? (directed? weighted? DAG?) and what do I need? (shortest path? connectivity? ordering? minimum total weight?).

You need…AlgorithmWhy this oneTime
Shortest path, unweighted (each edge = 1 hop) BFS Explores by distance layers — first arrival = fewest edges O(V+E)
Shortest path, non-negative weights Dijkstra Greedy: once a vertex is finalized, its distance is final O((V+E) log V)
Shortest path, negative weights OK Bellman-Ford Relaxes all edges V−1 times; detects negative cycles O(VE)
Is there a cycle? / topological order? DFS Back edge to gray vertex = cycle; finish times → topo sort O(V+E)
Dependency ordering (courses, build steps) Topological sort Only works on a DAG; if cycle exists → impossible O(V+E)
Connect all vertices, minimum total edge weight Kruskal / Prim Both find an MST; differ in how they grow the tree O(E log V)
Are two vertices connected? / count components? BFS or DFS Run from each unvisited vertex; each run = one component O(V+E)
Dynamic connectivity while adding edges Union-Find Kruskal uses this to check “same component?” in O(α(n)) O(m·α(n))
Real-world mapping: GPS with equal road segments → BFS. GPS with distances → Dijkstra. Flight prices with possible rebates (negative) → Bellman-Ford. Connect cities with minimum cable cost → MST. Course prerequisites → topological sort.
Exam pattern: many questions are not “run BFS” directly — they ask a variant (distance between two sets of vertices, path with color changes, “is edge e in some MST?”). The trick is usually a reduction: build a new graph G′, then run a standard algorithm. See §6.8.

6.1 Definitions & Representations

Graph \(G=(V,E)\): vertices + edges. Directed or undirected.

Degree — # incident edges.

Path — sequence of connected vertices; simple if no repeats.

Cycle — path returning to start.

Connected — path between every pair (undirected).

Connected component — maximal connected subgraph.

Tree — connected, acyclic, undirected (\(|E|=|V|-1\)).

DAG — directed acyclic graph.

6.2 BFS — Breadth-First Search

Problem it solves: explore all vertices reachable from a source \(s\), processing them in order of increasing distance (number of edges). On unweighted graphs, this gives the shortest path from \(s\) to every reachable vertex.

Idea: explore the graph in layers — all vertices at distance 1, then distance 2, etc. Uses a FIFO queue (first discovered, first explored).

What BFS is good for

ProblemWhy BFS?
Unweighted shortest path from \(s\)First time you reach \(v\), you used the fewest edges. \(dist[v]\) = hop count.
Reachability / connectivityVisits exactly the vertices reachable from \(s\). Run from each unvisited vertex to list all components.
Bipartite test (2-coloring)BFS levels alternate — if an edge connects same color, graph is not bipartite.
Level-order traversalSame wave idea as BFS on trees — process by depth from root.

When not to use BFS

  • Weighted shortest path → use Dijkstra (non-negative) or Bellman-Ford (negative OK).
  • Cycle detection in directed graphs → prefer DFS (back edge is easier to spot).
  • Topological order → DFS on a DAG, not BFS.
Exam mnemonic: “shortest path, all edges cost 1” → BFS. Complexity always O(V + E) with adjacency list — each vertex and edge touched once.

pseudocode 
BFS(G, s):
    color all white, dist = infinity
    dist[s] = 0
    enqueue(s)
    while Q not empty:
        u = dequeue()
        for v in Adj[u]:

                parent[v] = u
                enqueue(v)

Time O(V + E) · Space O(V) (queue + color/dist arrays).

BFS tree: edges used when first discovering a white vertex form a tree. Not every shortest path is a tree edge — but tree distance equals shortest distance. Exams sometimes ask how many distinct BFS trees exist (depends on tie-breaking when enqueueing neighbors).

6.3 DFS — Depth-First Search

Problem it solves: systematically explore a graph to answer structural questions — cycles, connectivity, dependency order, ancestor/descendant relationships — not necessarily shortest paths.

Idea: from a vertex, go as deep as possible along one path, then backtrack. Uses a stack (explicit) or recursion (call stack). Records discovery \(d[v]\) and finish \(f[v]\) timestamps.

What DFS is good for

ProblemWhy DFS?
Cycle detection (directed)A back edge to a gray (active) ancestor means a cycle exists.
Topological sortOn a DAG: output vertices in decreasing finish time (or prepend when finished).
Connected componentsEach DFS tree in the forest = one component (undirected).
Strongly connected componentsTwo-pass DFS (Kosaraju) or Tarjan — advanced, same \(O(V+E)\) spirit.
Path existence / mazeExplores all reachable vertices; does not guarantee shortest path.
Parenthesis theorem / intervalsNested \( [d[v], f[v]] \) intervals classify edge types — common proof question.

Edge types (directed graph)

Edge (u → v)Meaning
Tree\(v\) white when explored from \(u\) — discovery edge in DFS forest.
Back\(v\) is gray (ancestor of \(u\)) — cycle in directed graph.
Forward\(v\) black, \(v\) descendant of \(u\), not a tree edge.
Cross\(v\) black, not ancestor/descendant — between different branches.

In an undirected graph DFS only has tree and back edges (back = to an already discovered ancestor).

Exam mnemonic: “cycles, topo sort, finish times, components” → DFS. Same O(V + E) time as BFS with adjacency list.

Cycle detection: in a directed graph, a back edge (to a gray node) ⟺ cycle. In undirected graphs, back edge to parent’s ancestor also indicates a cycle.
White-path theorem: \(v\) is a descendant of \(u\) in the DFS forest ⟺ at time \(d[u]\) there is an all-white path \(u\rightsquigarrow v\).

6.4 Topological Sort

Problem it solves: given a DAG (directed acyclic graph), produce a linear order of vertices so that for every edge \((u,v)\), vertex \(u\) appears before \(v\). If there is a cycle, no valid order exists.

Why it exists: models dependencies — you cannot start task \(v\) until all prerequisites \(u\) are done. Examples: course prerequisites, build systems (compile before link), package installation order.

Algorithm (DFS-based): run DFS; when a vertex turns black (finished), prepend it to the output list. Equivalently: output in decreasing finish time \(f[v]\).

Multiple valid orders: a DAG may have many topological sorts — any order respecting all edges is valid. DFS finish-time order is one canonical choice.
Cycle = failure: if DFS finds a back edge, the graph is not a DAG and topological sort is impossible. Always check for cycles first.

Longest path in a DAG (exam variant)

On a DAG, the longest path from \(s\) to \(t\) can be found by: (1) topological sort, (2) process vertices in topo order, (3) relax edges like Dijkstra but take max instead of min. Time O(V + E). Works because topo order guarantees all predecessors of \(v\) are processed before \(v\).

6.5 Minimum Spanning Tree (MST)

What problem does MST solve?

You have a connected, weighted, undirected graph. You want to connect all vertices using exactly \(|V|-1\) edges (a tree) such that the total weight is minimized. Real-world: lay fiber-optic cables between cities at minimum cost; connect computers in a network; design a pipeline system.

Spanning tree vs MST — know the difference

ConceptDefinitionCheck
Spanning tree Subgraph that connects all \(V\) vertices, has exactly \(|V|-1\) edges, and has no cycles Connected + acyclic + \(|E_T|=|V|-1\)
NOT a spanning tree Has a cycle, or is disconnected, or has wrong edge count Any cycle ⟹ not a tree; missing vertex ⟹ not spanning
MST A spanning tree whose sum of edge weights is minimum among all spanning trees Compare total weight \(w(T)\) across all spanning trees

The two key properties (memorize for proofs)

Cut property: for any partition of \(V\) into \(S\) and \(V\setminus S\), the lightest edge crossing the cut belongs to some MST.
Intuition: to connect the two sides, you must use a crossing edge — pick the cheapest one.
Cycle property: in any cycle, the heaviest edge (strictly heaviest if weights are distinct) is never in an MST.
Intuition: you can always swap it for a lighter edge on the cycle and get a cheaper tree.

Edge membership — exam favorites

QuestionAnswer / test
Is \(e_{\min}\) (unique lightest edge) in every MST? Yes. Proof by contradiction: if some MST \(T\) lacks \(e_{\min}\), then \(T \cup \{e_{\min}\}\) has a cycle; remove the cycle's max edge (heavier than \(e_{\min}\)) → cheaper spanning tree. Contradiction.
Can \(e_{\max}\) be in an MST? Yes — e.g. if \(e_{\max}\) is the only edge to a leaf vertex (bridge to an isolated part), every spanning tree must include it.
Can there be multiple MSTs? Yes when weights tie. If all weights are distinct, the MST is unique (proof: swap argument on two different MSTs gives contradiction).
Is edge \(e=(u,v)\) in some MST? \(e\) is not the unique heaviest edge in any cycle. Build \(G'\) with edges lighter than \(e\); run BFS/DFS from \(u\). If \(v\) is reachable → cycle exists where \(e\) is heaviest → \(e\) is safe. Else → \(e\) is in every MST.
Is edge \(e\) in every MST? Same test but \(G'\) uses edges with weight \(\le w(e)\), excluding \(e\) itself. If \(u \leadsto v\) in \(G'\) → there is a cycle with an equal-or-lighter alternative → \(e\) not mandatory.

Both edge-membership tests run in O(V + E) — build \(G'\) + one BFS/DFS.

Kruskal — edge-centric

Strategy: sort edges globally, add cheapest first if it doesn't create a cycle.

  1. Sort all edges by weight ascending.
  2. For each edge \((u,v)\) in order: if \(u\) and \(v\) are in different components, add the edge and union the components.
  3. Stop after \(|V|-1\) edges.

Data structure: Union-Find (disjoint sets with union-by-rank + path compression).

StepTime
Sort edgesO(E log E)
\(E\) union-find opsO(E · α(V))
TotalO(E log V)

Prefer Kruskal when: graph is sparse, you think in terms of edges, or you already have Union-Find.

Prim — vertex-centric

Strategy: grow one tree from a root; always attach the closest outside vertex.

  1. Start from arbitrary root \(r\); \(key[v]\) = min edge weight from \(v\) to the current tree (∞ if not adjacent).
  2. Repeatedly pick vertex \(u\) with minimum \(key[u]\), add it to the tree.
  3. For each neighbor \(v\) of \(u\) outside the tree: if \(w(u,v) < key[v]\), update \(key[v]\) and parent.
  4. Stop when all vertices are in the tree.

Data structure: min-heap (binary heap) for extract-min + decrease-key.

StepTime
Init keysO(V)
\(V\) extract-min + up to \(E\) decrease-keyO(E log V)
TotalO(E log V)

Prefer Prim when: graph is dense, or you naturally grow from a known starting point.

Kruskal vs Prim — decision guide

ConditionBetter choiceResulting complexity
Standard implementationEither — both O(E log V)Equivalent asymptotically
Edge weights in [1, V] (integers)Kruskal with counting sort on edgesO(E · α(V))
Edge weights in [1, W], \(W\) constantPrim with bucket PQ (array of linked lists per weight)O(E)
Some edges already exist (weight 0)Either MST algorithm — zero-weight edges are taken firstModel as MST on full graph

Modeling real problems as MST

Example — connect cities with existing cables: build complete graph on cities; existing connections get weight 0, new connections get cost \(c(u,v) > 0\). Run MST. The result uses all zero-weight (existing) edges that don't form a cycle, plus minimum-cost new edges to connect everything. Total cost = sum of weights of new edges in the MST.

Exam tip: once you recognize “connect all nodes, minimize total cost,” write “reduce to MST” and cite Kruskal or Prim as a black box — then analyze the reduction + MST runtime.

6.6 Dijkstra — Single-Source Shortest Paths

Problem it solves: find shortest paths from source \(s\) to all vertices when edge weights are non-negative. Generalizes BFS to weighted graphs.

Strategy: greedy — repeatedly finalize the closest unvisited vertex, then relax each outgoing edge \((u,v)\): if \(dist[u]+w(u,v) < dist[v]\), update \(dist[v]\).

pseudocode 
Dijkstra(G, w, s):
    dist[s] = 0, others = infinity
    PQ = all vertices
    while PQ not empty:
        u = extractMin(PQ)
        for v in Adj[u]:
            if dist[u] + w(u,v) < dist[v]:
                dist[v] = dist[u] + w(u,v)
                decreaseKey(v)
Binary heapO((V+E) log V)
Fibonacci heapO(E + V log V)
Fails with negative edges! Once a vertex is finalized it's never revisited. Use Bellman-Ford (O(VE)) for negative weights.
BFS vs Dijkstra: BFS = Dijkstra with all weights 1. On unweighted graphs, prefer BFS.

6.7 Bellman-Ford — Negative Weights OK

Problem it solves: single-source shortest paths when edge weights can be negative. Also detects negative-weight cycles reachable from \(s\) (then shortest paths are undefined).

Why Dijkstra fails: Dijkstra assumes that once a vertex is extracted from the priority queue, its distance is final. A later negative edge could shorten it — Dijkstra never revisits.

pseudocode 
Bellman-Ford(G, w, s):
    dist[s] = 0, others = infinity
    repeat |V| - 1 times:
        for each edge (u,v) in E:
            if dist[u] + w(u,v) < dist[v]:
                dist[v] = dist[u] + w(u,v)
    for each edge (u,v):
        if dist[u] + w(u,v) < dist[v]:
            return "negative cycle"
    return dist

Time O(VE) · Space O(V).

Intuition: after \(k\) rounds, \(dist[v]\) is the shortest path using at most \(k\) edges. After \(|V|-1\) rounds, all simple shortest paths are found (a simple path has at most \(|V|-1\) edges).
ScenarioUse
All weights ≥ 0Dijkstra — faster
Negative weights, no negative cycleBellman-Ford
Need to detect negative cycleBellman-Ford (extra pass)
All-pairs shortest paths (small V)Floyd-Warshall O(V³) — beyond scope but know the name

6.8 Graph Reductions — Exam Pattern

Many exam questions don't ask you to run BFS/DFS directly — they describe a variant. The standard approach: reduce the new problem to a known one by building graph \(G'\), then run the standard algorithm.

Pattern 1: Super-source / super-sink

Problem: find shortest distance between two sets of vertices \(V_1\) and \(V_2\) (not just two vertices).

Reduction: add dummy vertex \(s\), connect \(s \to u\) for all \(u \in V_1\) (weight 0); add dummy \(t\), connect \(v \to t\) for all \(v \in V_2\) (weight 0). Run BFS/Dijkstra from \(s\). Answer = \(dist[t] - 2\) (subtract the two dummy hops).

Time: building \(G'\) is O(V + E); BFS on \(G'\) is O(V + E).

Pattern 2: Vertex splitting (track a state)

Problem: is there a path from \(s\) to \(t\) with exactly \(k\) “color changes” (edges between differently-colored vertices)?

Reduction: for each vertex \(v\), create two copies \(v_0\) and \(v_1\) in \(G'\):

  • Same-color edge \((u,v)\) → add \((u_0, v_0)\) and \((u_1, v_1)\)
  • Different-color edge \((u,v)\) → add \((u_0, v_1)\) only (one color flip)

Run DFS/BFS from \(s_0\) to check reachability of \(t_1\) (exactly one color change). Generalizes to \(c\) flips with \(c+1\) copies per vertex.

Pattern 3: Reverse graph / transpose

Problem: “can every vertex reach \(t\)?” or “vertices from which \(t\) is reachable?”

Reduction: reverse all edges to get \(G^T\), run BFS/DFS from \(t\) in \(G^T\). Vertices visited = vertices that can reach \(t\) in the original graph.

Pattern 4: Reduce to MST

“Connect all nodes at minimum cost, some connections already exist” → assign weight 0 to existing edges, positive cost to new ones → run Kruskal/Prim on the complete graph.

Pattern 5: Reduce to edge-in-MST test

“Is edge \(e\) in some/every MST?” → build subgraph with only lighter (or lighter-or-equal) edges, run BFS/DFS to check if endpoints are connected without \(e\). See §6.5.

Always write the reduction explicitly in exams: (1) define \(G'=(V',E')\), (2) state the claim (“path exists in \(G\) ⟺ path exists in \(G'\)”), (3) run known algorithm, (4) analyze total time as O(build) + O(algorithm).

6.9 Exam Tips & Conclusions

Before you code — ask these 4 questions

  1. Directed or undirected? Affects edge types, cycle detection, and whether to use both directions in adjacency list.
  2. Weighted or unweighted? Unweighted → BFS. Non-negative weights → Dijkstra. Negative OK → Bellman-Ford.
  3. What is the output? Distance? Path? Yes/no (connectivity)? Ordering? Minimum total weight?
  4. Is this a standard problem or a variant? If variant → look for a reduction (super-source, vertex split, reverse graph, MST).

Common mistakes (from practice sessions & past exams)

  • Using BFS for weighted graphs — BFS only gives shortest hop count, not minimum weight.
  • Using Dijkstra with negative edges — greedy finalization breaks; use Bellman-Ford.
  • Topological sort on a graph with cycles — always check for back edges first.
  • Confusing spanning tree with MST — any acyclic connected subgraph with \(|V|-1\) edges is a spanning tree; MST additionally minimizes total weight.
  • Forgetting Union-Find in Kruskal analysis — mention α(V) per operation, not O(1) amortized without justification.
  • Not specifying data structure — “run Prim” is incomplete; say “Prim with binary min-heap” and derive O(E log V).
  • Reduction without proof sketch — write one sentence why the reduction preserves the answer.
  • Off-by-one in super-source reduction — remember to subtract dummy edge count from final distance.

Proof techniques that appear repeatedly

Claim typeTechnique
“Lightest edge is in every MST”Contradiction: add edge → cycle → swap heavier edge → cheaper tree
“MST is unique” (distinct weights)Contradiction: two MSTs differ on edge \(e\) → cycle with lighter swap
“Edge \(e\) safe to add”Cut property — \(e\) is lightest crossing some cut
“Edge \(e\) never in MST”Cycle property — \(e\) is heaviest in some cycle
“BFS finds shortest path”Induction on distance layers — first visit at layer \(d\) means shortest path has length \(d\)

How to write algorithm answers in exams

  1. State the reduction (if any) — define \(G'\) explicitly.
  2. Name the algorithm and the data structure (heap, Union-Find, queue).
  3. Write main logic — 5–10 lines of pseudocode; skip boilerplate init unless asked.
  4. Analyze runtime — separate build time from algorithm time; state the dominant term.
  5. Handle edge cases in one line: disconnected graph, empty graph, negative cycle.
Black-box usage: in many exam questions you can treat MST, BFS, or Dijkstra as a subroutine (“run Kruskal on \(G\)”) and only analyze the reduction + black-box cost. Don't re-derive Kruskal unless the question asks for it.
When stuck: draw a small example graph (4–5 vertices), trace the algorithm by hand, then generalize. Most graph exam questions test understanding of what the algorithm preserves, not tricky implementation details.
